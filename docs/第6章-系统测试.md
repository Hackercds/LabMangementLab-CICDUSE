# 6 系统测试

系统测试是软件交付生命周期中验证功能实现与量化非功能属性的关键环节。本章在概述测试环境与方法论之后，首先简要介绍单元测试与功能测试的结果，随后将重点置于性能测试——从测试目标与度量体系、环境与基础设施、场景设计、结果分析到综合评定，系统性地评估系统在高并发条件下的响应延迟、吞吐量、资源利用效率与数据一致性保障能力。

### 6.1.2 测试体系设计

本研究的测试策略遵循 IEEE 829-2008《软件测试文档标准》中推荐的层级化测试思想，构建单元测试、功能测试与性能测试三层的递进式验证体系[1]。

**单元测试层**以 JUnit 5 为测试运行器、Mockito 为依赖模拟框架，运用等价类划分与边界值分析等经典测试设计技术，对预约冲突检测的时间区间重叠判定、设备生命周期状态机转换、耗材库存扣减与预警阈值判断以及 JWT 令牌的签发与校验四类高风险业务逻辑进行白盒验证[2]。

**功能测试层**采用黑盒测试方法论，按照需求规格说明书中定义的功能边界，以等价类划分法设计正面测试用例（验证合法输入下的预期输出）和负面测试用例（验证系统在非法输入下的防御能力），逐模块验证 API 接口的响应码、响应体结构以及数据库侧状态的最终一致性[3]。

**性能测试层**是本章的核心重点。采用负载测试与并发竞态测试相配合的策略——负载测试通过构建典型业务混合流量，在较长时间窗口内持续加压，考察系统在稳态下的吞吐量-响应时间关系；并发竞态测试则针对共享资源的临界访问设计集中的竞争场景，以验证数据库锁机制在高冲突条件下的正确性及时效性。测试期间借助 Prometheus + Grafana 监控体系实时采集 JVM 内存与 GC 行为、CPU 利用率、数据库连接池状态等运行时指标，为性能瓶颈的识别与量化分析提供实证数据[4]。

### 6.1.3 关键测试与监控组件描述

**（1）Apache JMeter**

JMeter 是 Apache 软件基金会旗下的开源负载测试工具，采用纯 Java 实现，支持多线程并发请求生成、参数化数据驱动（CSV Data Set Config）、断言验证（Response Assertion、Duration Assertion）以及多种性能监听器的聚合报告输出[5]。本研究选用 JMeter 6.0 版本基于以下考量：其一，Java 技术栈的工具链一致性避免了跨语言测试工具的协议适配偏差；其二，JMeter 提供的 Constant Throughput Timer（固定吞吐量定时器）能够精确控制目标请求速率，使测试结果在不同轮次之间具有可比性；其三，SetupThreadGroup 与 PostThreadGroup 机制支持测试前置数据准备与后置数据清理的自动化，确保每次测试运行的初始状态一致。

**（2）Prometheus + Grafana 监控体系**

Prometheus 是 Cloud Native Computing Foundation（CNCF）毕业项目，采用基于拉取（pull-based）的时序数据采集模型，通过多维标签（label）对指标进行分类和聚合查询[6]。Grafana 则是与之配套的可视化平台，支持 PromQL 查询语言，能够以仪表盘形式实时展示 JVM 堆内存、GC 停顿、CPU 利用率、数据库连接池等运行时指标的时序变化。

本研究在后端引入 Micrometer 指标门面与 Prometheus Registry 适配器，通过 `/actuator/prometheus` 端点暴露 JVM 指标（内存池使用量、GC 次数与耗时、线程数）、HikariCP 连接池指标（活跃/空闲/等待连接数）以及 HTTP 请求指标。选取该监控方案的理由如下：首先，Prometheus 的拉取模型对被测应用的侵入性低，不会因监控代理的数据推送而额外消耗网络与 CPU 资源；其次，PromQL 的聚合与分位数计算能力（如 `histogram_quantile`）为 P95、P99 等尾延迟指标的获取提供了原生支持；再次，Grafana 仪表盘的时序面板使得 GC 行为、内存变化趋势等难以从单次快照中获知的动态特征变得直观可读。

**（3）G1GC 垃圾回收器**

G1（Garbage-First）GC 是 JDK 9 以来的默认垃圾回收器，其设计思想是将堆划分为多个等大小的 Region，通过并发标记与增量式回收（Mixed GC），在满足用户设定的暂停时间目标（`-XX:MaxGCPauseMillis`）的前提下优先回收垃圾最多的 Region[7]。本研究选择 G1GC 并设置最大暂停目标为 200ms，基于以下理由：该配置能够将单次 GC 停顿控制在可预测的上限以内，避免因 Full GC 导致的长时间 STW（Stop The World）停顿对 95% 以上请求的响应延迟产生不可接受的冲击；同时 G1GC 的区域化回收策略在 1GB 以内的堆配置下已被广泛验证为高效且稳定的选择。

**（4）HikariCP 数据库连接池**

HikariCP 是 Spring Boot 2.x/3.x 的默认数据库连接池实现，以其极低的连接创建开销和精简的字节码体积著称[8]。连接池的核心价值在于复用已创建的数据库连接，避免每次业务请求都经过 TCP 三次握手、MySQL 认证握手及线程创建的昂贵开销。本研究将 `maximumPoolSize` 设置为 20、`minimumIdle` 设置为 5，理由是：在测试环境单机部署条件下，MySQL 默认 `max_connections` 为 151，20 个连接上限在保证并发处理能力的同时，不会对数据库连接资源造成过度占用；同时后端与 MySQL 通过 Docker 内部桥接网络通信（局域网延迟 < 1ms），单个连接的吞吐能力较高，20 个连接足以应对 500 并发的稳态负载，过多的连接配置反而会增加 MySQL 服务端线程调度开销与内存占用。

## 6.2 单元测试

单元测试聚焦后端核心业务逻辑中正确性要求最高的四个高风险模块，基于 JUnit 5 + Mockito 框架，以等价类划分与边界值分析等方法进行白盒验证[2]。四个测试类共计 17 个测试用例，全部通过。

### 6.2.1 预约冲突检测算法测试

被测类 `ReservationService` 的冲突检测逻辑基于 Allen 时间区间关系代数[9]，判定两个预约时段是否存在重叠。测试用例分别选取了区间相邻、部分重叠、完全包含、被完全包含及有间隙五种拓扑关系，覆盖了时间区间间所有可能的非平凡关系类型。

**表 6-2 预约冲突检测单元测试用例**

| 用例 | 已有预约 | 新预约 | 区间拓扑关系 | 预期冲突计数 | 实测 |
|------|----------|--------|-------------|-------------|------|
| TC-R-01 | 08:00-10:00 | 10:00-12:00 | `meets`（邻接） | 0 | 通过 |
| TC-R-02 | 08:00-10:00 | 09:00-11:00 | `overlaps`（部分重叠） | 1 | 通过 |
| TC-R-03 | 08:00-12:00 | 09:00-10:00 | `during`（被包含） | 1 | 通过 |
| TC-R-04 | 09:00-11:00 | 08:00-12:00 | `contains`（包含） | 1 | 通过 |
| TC-R-05 | 08:00-10:00 | 11:00-12:00 | `before`（有间隙） | 0 | 通过 |

TC-R-01 与 TC-R-05 区别了"邻接不冲突"与"有间隙不冲突"两种非重叠子类型，TC-R-02 至 TC-R-04 覆盖了三种重叠子类型，组合覆盖完备。

### 6.2.3 设备状态转换测试

设备实体的生命周期采用有限状态机（FSM）建模，包含 NORMAL（可借用）、BORROWED（已借出）、MAINTENANCE（维护中）三种状态。借用操作仅允许 NORMAL→BORROWED 迁移，归还操作仅允许 BORROWED→NORMAL 迁移。依据 FSM 测试理论[10]，每条合法迁移边及每条非法迁移边均需逐一验证。

**表 6-4 设备状态转换单元测试用例**

| 用例 | 初始状态 | 操作 | 预期迁移 | 预期结果 | 实测 |
|------|----------|------|----------|----------|------|
| TC-D-01 | NORMAL | borrow | NORMAL→BORROWED | 成功，写入借用历史 | 通过 |
| TC-D-02 | BORROWED | borrow | — | 抛出异常，拒绝 | 通过 |
| TC-D-03 | MAINTENANCE | borrow | — | 抛出异常，拒绝 | 通过 |
| TC-D-04 | BORROWED | return | BORROWED→NORMAL | 成功，更新归还时间 | 通过 |
| TC-D-05 | NORMAL | return | — | 抛出异常，拒绝 | 通过 |

五个用例穷尽了 FSM 中两条合法迁移边及三条非法迁移路径，确保了设备在任何状态下对借用和归还操作的响应行为确定且符合业务规则。

## 6.4 性能测试

本章的测试重点在于性能评估。以下从测试目标、环境配置、场景设计、结果分析到综合评定，系统性地展开。

### 6.4.1 测试目标与度量体系

性能测试旨在回答以下三个核心问题：（1）系统在目标并发量级下是否能够维持稳定服务，不出现崩溃、连接耗尽或响应超时；（2）对于涉及共享资源争用的写入操作（预约创建、设备借用、审批），数据库锁机制能否在保证数据一致性的前提下将响应时间控制在可接受范围内；（3）在长时间稳态负载下，系统资源是否存在持续退化趋势（如内存泄漏、GC 频率逐步升高）。

参照企业级服务质量（Service Level Agreement, SLA）的度量规范，本研究在传统平均响应时间指标之外，引入统计分位数与运行时指标以构建更为全面的性能评价基准[12]：

**（1）响应延迟分位数**：以 JMeter 采集的样本为总体，按升序排列后取第 50 百分位数（P50/中位数）、第 95 百分位数（P95）及第 99 百分位数（P99）。其中 P50 反映典型用户体验，P95 和 P99 则衡量长尾延迟（tail latency）——即少数慢请求偏离中位数的程度。Dean 与 Barroso[13]指出，在大规模系统中，P99 乃至 P99.9 的尾部延迟往往比平均值更具工程意义，因为一个需要汇聚多个 API 调用的用户操作，其体验取决于所有调用中最慢的那一个。

**（2）吞吐量**：以每秒处理请求数（Requests Per Second, RPS）为单位，通过 Aggregate Report 中的实际吞吐量数据与 JMeter Constant Throughput Timer 设定的目标值进行对比，评估系统在给定负载下的流量处理能力。

**（3）GC 度量**：通过 Prometheus 采集 JVM G1GC 的 `jvm_gc_pause_seconds` 指标，分别计算 Young GC 的平均停顿时间、P99 停顿时间及最大停顿时间，同时监控 Full GC 的发生次数。GC 停顿直接影响请求的尾部延迟——一次超过数十毫秒的 STW 停顿可导致该窗口内所有未完成请求的响应时间同步飙升[14]。

**（4）资源利用率**：通过 Prometheus + Grafana 采集宿主机 CPU 利用率（`process_cpu_usage`）、JVM 堆内存使用率（`jvm_memory_used_bytes / jvm_memory_max_bytes`）以及 HikariCP 数据库连接池的活跃连接数（`hikaricp_connections_active`）等时序数据，评估资源使用的效率与余量。

### 6.4.2 测试环境与基础设施

**硬件环境**：宿主机处理器为 AMD Ryzen 7 PRO-6850H（8 核心 16 线程，Zen 3+ 架构，最高加速频率 4.7GHz）；内存为 16GB LPDDR5，频率 5,600MHz；存储为华为坤灵 eKitStor Xtreme 200E NVMe SSD，容量 1TB，PCIe 4.0×4 接口，标称顺序读取速度 ≥7,000MB/s。

LPDDR5 5,600MHz 的高带宽（较 DDR4-3200 提升约 75%）为 JVM 堆内存的频繁对象分配与回收提供了充裕的数据吞吐能力。在 Spring Boot 处理每个 HTTP 请求的过程中，需要创建诸多短生命周期实例（`HttpServletRequest`、`HttpServletResponse`、Controller 参数对象、MyBatis 映射对象等），这些对象在请求结束后即成为垃圾。LPDDR5 的高带宽使得频繁的堆内存分配-回收循环能够在纳秒级完成，降低了对象分配在响应路径中的时间占比。

NVMe SSD 的低延迟与高 IOPS 特性则确保了 MySQL InnoDB 事务提交（redo log flush）与随机读写的响应速度，在并发写入场景下对锁持有时间的缩短具有直接贡献。MySQL InnoDB 的每次事务提交均涉及 redo log 的持久化写盘（`innodb_flush_log_at_trx_commit=1`），一个 0.1ms 的 NVMe 写入延迟相比典型 SATA SSD 的 0.5~1ms，意味着每次事务提交可节省约 0.4~0.9ms。在 260 线程并发竞争的预约场景中，260 个事务串行提交累积节约的时间可达约 100~230ms——事务提交越快，锁释放越快，后续排队请求的等待时间越短。

**软件环境**：测试在单机 Docker 环境下执行，后端（Spring Boot）、MySQL 8.0、Redis 7 以容器方式共享宿主机资源。后端 JVM 启动参数为 `-Xms512m -Xmx1g -XX:+UseG1GC -XX:MaxGCPauseMillis=200`。选择 G1GC 的理由已在 6.1.3 节阐明。堆内存上限设为 1GB 的考量在于：在容器化部署中，JVM 堆内存是容器配额的主要去向，过大堆配置既浪费资源又可能因操作系统级的内存换页引发性能退化[15]；以 1GB 的小堆配置通过高负载测试，能够证明系统在有限资源条件下的内存效率是合格的。

HikariCP 连接池配置为 `maximumPoolSize=20`，`minimumIdle=5`。该配置的合理性在于：后端与 MySQL 通过 Docker 内部桥接网络通信（局域网延迟 < 1ms），单个数据库连接的吞吐能力较高，20 个连接上限足以应对 500 并发的稳态负载，过多的连接配置反而会增加 MySQL 服务端线程调度开销与内存占用[8]。

JMeter 6.0 运行于同一宿主机（非 GUI 模式），通过 Aggregate Report 与 Aggregate Graph 收集测试数据，测试期间同步开启 Prometheus + Grafana 采集运行时指标。

| 指标 | 数值 | 判定 |

一个值得关注的发现是：尽管宿主机配备 8 核心 16 线程的 AMD R7 PRO-6850H 处理器，但在约 1,300 RPS 的稳态高负载下，系统实际仅占用约 1 个物理核心（利用率约 90%），其余 7 核基本处于空闲状态。这一现象可以从以下几个方面加以分析。

其一，Tomcat 默认线程池（`max-threads=200`）的线程调度模型本质上是将请求分布到可用处理器上的，但在本测试中，每个请求的平均 CPU 时间极短——响应时间约 84ms 中的大部分消耗在等待数据库 I/O（查询执行、结果集传输）和网络 I/O 上，而非 CPU 计算，单个核心即可在时间片轮转中覆盖全部并发请求的 CPU 需求。这符合 I/O 密集型应用的典型特征：CPU 并非瓶颈，请求的响应时间主要由 I/O 等待构成。

其二，从 Little 定律[16]的角度分析：系统平均并发请求数 $L = \lambda \times W = 1,289 \text{ RPS} \times 0.084\text{s} \approx 108$。这 108 个同时在途的请求中，绝大多数处于等待数据库返回结果或网络传输的状态（非 RUNNING 状态），真正竞争 CPU 时间片的线程数远小于 108，因此单个核心足以调度。

其三，这一现象并不意味着多核资源被浪费——相反，它表明系统在当前负载水平下具有极高的横向扩展潜力。单核处理约 1,300 RPS 即达到 90% 利用率，意味着在需要承载更高吞吐量时，只需增加 Tomcat 工作线程数并利用其余核心，理论上 8 核可线性扩展至约 10,000 RPS 的处理能力（在数据库和内存不成为瓶颈的前提下）。此外，单核接近满载但未触顶（仍有约 10% 余量）的状态恰恰说明当前配置的线程数与负载水平匹配良好，既未因线程数过少而积压请求，也未因线程数过多而引发过度的上下文切换开销。

**GC 行为**如表 6-17 所示。10 分钟内共发生 47 次 Young GC，无 Full GC。

**表 6-17 场景 B GC 行为统计**

|------|------|------|

| 业务成功率 | 100% | 0.38%（1/260） |

260 个请求在 2 秒窗口内集中到达，`SELECT ... FOR UPDATE` 悲观锁将并发请求序列化为排队处理。P50 为 742ms，P99 为 2,618ms，最大值 2,912ms，均在预设的 3 秒响应断言范围内，无请求超时或连接池耗尽。从业务语义角度，仅 1 个请求成功创建预约（PENDING 状态），其余 259 个请求因冲突检测返回拒绝——这恰好验证了悲观锁在极端竞争下的序列化正确性。

场景 C 的 P99（2,618ms）较场景 B/G 的稳态负载 P99（< 150ms）高出约 17 倍，其根本原因并非应用代码层面的性能瓶颈，而是数据库锁的排队效应——260 个线程对同一行记录的竞争形成了严格的先入先出（FIFO）顺序队列，第 $k$ 个线程必须等待前 $k-1$ 个事务全部提交后方能获取锁。这种性能代价是为保证数据一致性而做出的必要权衡，在 OLTP 系统的并发控制理论中已有充分论述[17]。在日常业务中，260 个用户同时争夺同一实验室同一时段的概率极低，该场景仅用于验证极端条件下的正确性边界，其响应延迟不代表系统的常规性能水平。

#### 6.4.4.4 场景 D：设备借用并发（260 线程）

260 个线程同时对同一设备发起借用请求，响应时间统计如表 6-20 所示。

**表 6-20 场景 D 响应时间统计**

| P95 | 198ms | 1,527ms |

| 错误率 | 0% | 0% |

P99 = 2,103ms，所有请求在 5 秒断言内完成。`selectByIdForUpdate` 确保了仅一个线程成功将设备从 NORMAL 更新为 BORROWED，其余 259 个请求因状态校验失败返回"设备不可借阅"，无死锁或重复更新。

| P95 | 182ms | 215ms | 936ms |

| 最大值 | 278ms | 312ms | 1,503ms |

审批操作 P99 = 1,274ms。首次审批成功将预约状态从 PENDING 更新为 APPROVED，其余线程因检测到状态已变更返回"已处理"。50 线程的锁竞争远低于 260 线程场景，P99 较场景 C 收窄约 51%，与排队线程数减少的预期一致。

500 个用户在 10 秒内完成加载，每个线程依次执行登录、仪表盘查询、预约概览查询和设备列表分页查询，模拟瞬时高并发登录与密集读取的压力特征。

### 6.4.5 性能测试综合评定

基于以上七个场景的测试数据，对系统性能做出如下评定：

**（1）响应延迟控制优异**：在约 1,300~1,560 RPS 的稳态读写混合负载下，各接口综合 P95 ≤ 120ms，综合 P99 ≤ 147ms，且 10 分钟持续运行期间未出现 P95/P99 的持续攀升趋势，排除了慢查询积累或资源泄漏的可能。写入操作（P95 = 142ms、P99 = 178ms）与读操作（P95 = 78~128ms）之间的延迟差距仅为 32~64ms，源于事务提交与冲突检测查询的固有开销，处于合理范围。高竞争并发场景（场景 C、D）下的 P99 升至 2~3s，系悲观锁序列化排队的必然结果，仅在 260 线程同时争用同一资源的极端条件下出现。

**（2）单核高效，扩展潜力充裕**：系统在 AMD R7 PRO-6850H（8C/16T）处理器上仅占用约 1 个核心（单核利用率 ~90%）即承载了约 1,560 RPS 的吞吐量，单核每毫秒约可处理 1.7 个请求。剩余 7 个核心提供了充裕的横向扩展余量——如需承载更高吞吐量，仅需增加 Tomcat 工作线程数并利用多核并行能力，理论上可获得近似线性的吞吐量提升。同时，单核 ~90% 的利用率处于"充分利用但未过载"的健康区间，避免了因 CPU 饱和导致的请求排队堆积。

**（3）内存效率显著**：在 500 并发高负载下，1GB 堆内存仅使用 28%~34%，稳态约 320MB。这意味着系统在以较低内存成本运行时仍保留了约 65% 的缓冲容量，为突增流量（如开学季集中预约、课表调整期间的密集访问）提供了充裕的安全余量，无 OOM（Out Of Memory）风险。低堆内存占用也得益于 Spring Boot 与 MyBatis-Plus 在对象分配方面的设计——请求作用域内的对象生命周期短、晋升至老年代的比例低，Young GC 即可高效回收。

**（4）GC 行为健康**：各场景下 Young GC P99 停顿 ≤ 32.1ms（目标 < 36ms），平均停顿 ≤ 14.2ms。整个测试期间 Full GC 零次，老年代晋升率低。单用户条件下 GC 停顿 < 10μs（约 0.01ms），对用户请求不构成可感知的影响。G1GC 在 1GB 低堆配置下表现稳定，Young GC 停顿与响应时间 P95/P99 之间的量级关系合理——GC 不是影响响应延迟的主要因素。

**（5）数据一致性保障有效**：在 260 线程并发竞态的三个场景（C、D、E）中，`SELECT ... FOR UPDATE` 悲观锁机制正确保障了临界资源的序列化访问。未观察到数据不一致、状态覆盖或重复分配的情况。唯一成功的写入操作严格对应第一个获取锁的线程，与悲观锁的先入先出语义一致。

**（6）硬件适配性分析**：测试结果与硬件配置之间存在合理的适配关系。LPDDR5 5,600MHz 的高带宽为 JVM 对象分配提供了低延迟的数据通路，支持了约 1,560 RPS 吞吐量下 Young GC 的快速执行（均值 14.2ms）。NVMe SSD 的低延迟（约 0.1ms 写入延迟）对写入场景的事务响应时间具有直接贡献——按每个事务提交节约 0.4~0.9ms（相较于 SATA SSD）估算，在 260 线程串行提交的预约竞态场景中，仅存储性能一项就为系统节约了约 100~230ms 的总排队时间。AMD R7 PRO-6850H 的单核高频率（4.7GHz 加速）为 BCrypt 和 JWT 签名等 CPU 密集型安全运算提供了足够算力，使得 500 线程瞬时并发登录场景下登录 P99 仍能控制在 482ms。若将系统部署于更低规格的硬件（如 SATA SSD、DDR4 内存），性能指标——尤其是写入密集型场景的尾部延迟——将相应升高，本文报告的测试结论需在所述硬件条件下加以理解。

## 6.5 测试结论

综合单元测试、功能测试与性能测试三个层次的全部验证数据，做出如下结论。

**功能正确性**：系统的十大功能模块、38 项功能测试用例全部通过。17 项单元测试从代码层面穷尽了冲突检测算法的时间区间拓扑覆盖（5 种 Allen 关系全遍历）、设备状态机的全部迁移路径（2 合法 + 3 非法）、库存预警的阈值临界判定（4 种"库存 vs. 阈值"关系）以及 JWT 令牌的防篡改校验（3 种签名验证路径）。上下两层测试结论互洽，未出现单元测试与功能测试结果矛盾的情况。

**性能可靠性**：在每接口约 260 RPS、总计约 1,300~1,560 RPS 的稳态读写混合负载下，各接口综合 P95 ≤ 120ms、P99 ≤ 147ms，全接口错误率为零，10 分钟持续运行未出现性能退化。在 260 线程极端并发竞态场景下，P99 ≤ 2,912ms（均在 3 秒响应断言范围内），悲观锁机制正确保障了数据一致性，未出现死锁、连接耗尽或响应超时等系统性故障。

**资源效率**：系统在 AMD R7 PRO-6850H（8C/16T）+ LPDDR5 5,600MHz + NVMe SSD 的硬件平台上，仅需约 1 个核心（~90% 利用率）及 320MB 堆内存（1GB 上限的 32%）即达到 1,560 RPS 的吞吐能力。单核每毫秒处理约 1.7 个请求。剩余 7 个 CPU 核心与约 700MB 堆内存裕度、以及 NVMe SSD 的高 IOPS 储备，为未来业务量的增长提供了无需硬件升级即可获取的扩展空间。

**工程合理性**：系统在测试方法论层面遵循 IEEE 829-2008 的层级化测试框架，在工具选型层面采用 G1GC + HikariCP + JMeter + Prometheus/Grafana 的企业级技术栈。测试数据表明：G1GC 在 1GB 低堆配置下实现了预期的低停顿特性（Young GC P99 < 36ms）；HikariCP 以 20 个连接上限支撑了 500 并发的稳态流量；Prometheus/Grafana 的实时指标采集体系为 GC 行为、内存趋势和连接池状态的定量分析提供了可追溯的数据基础；以 260 RPS/接口、500 并发的超出标称负载进行压力测试，验证了系统的冗余承载能力。

综上所述，经过系统性的测试验证，本实验室管理系统在功能完备性、业务可靠性和并发性能三个维度均达到设计规格的各项要求，具备在高校实验室管理场景中实际部署与推广应用的技术条件。
